<objective>
Wire on-demand ISR revalidation so that publishing any document in Sanity Studio invalidates the corresponding Next.js cache tag within seconds, replacing the 30-minute polling interval as the primary cache invalidation mechanism.

Purpose: Content editors currently wait up to 30 minutes to see published changes. This phase makes content updates visible within seconds by connecting Sanity webhooks to Next.js `revalidateTag()`. The existing `export const revalidate = 1800` stays as a safety-net fallback.

Output: Tagged fetches on all loaders, `unstable_cache` removed from layout, webhook Route Handler at `/api/revalidate` with HMAC-SHA256 signature verification.
</objective>

Create a new file at `frontend/src/app/api/revalidate/route.ts`. This is an App Router Route Handler that receives Sanity webhook POSTs and calls `revalidateTag()`.

**File content:**

```typescript
import { revalidateTag } from "next/cache";
import type { NextRequest } from "next/server";
import { parseBody } from "next-sanity/webhook";
import { webhookSecret } from "@/sanity/env";

// Map Sanity document _type to cache tag (per D-01, D-05, D-06, D-07)
const TYPE_TAG_MAP: Record<string, string> = {
  homePage: "homePage",
  contactPage: "contactPage",
  galleryPage: "galleryPage",
  trainingPage: "trainingPage",
  trainingProgram: "trainingProgram",
  globalSettings: "global",
  mainMenu: "menu",
};

export async function POST(req: NextRequest): Promise<Response> {
  // parseBody reads raw body text, verifies HMAC-SHA256, then JSON.parses
  // Do NOT call req.json() before this — it would consume the stream
  const { body, isValidSignature } = await parseBody<{ _type: string }>(
    req,
    webhookSecret,
  );

  // Per D-08: HTTP status codes only, no detail in response body
  // Per Pitfall 4: isValidSignature is null when no secret — treat as failure
  if (isValidSignature !== true) {
    console.warn("[revalidate] Invalid webhook signature");
    return new Response(null, { status: 401 });
  }

  if (!body?._type) {
    console.warn("[revalidate] Webhook body missing _type");
    return new Response(null, { status: 400 });
  }

  const tag = TYPE_TAG_MAP[body._type];

  if (!tag) {
    // Unknown type — not an error, just nothing to revalidate (per D-09)
    console.log(`[revalidate] Unhandled _type: ${body._type} — no-op`);
    return new Response(null, { status: 200 });
  }

  // Per research: Next.js 16 requires { expire: 0 } for immediate expiry
  // Single-arg revalidateTag(tag) is deprecated in Next.js 16
  revalidateTag(tag, { expire: 0 });
  console.log(`[revalidate] tag='${tag}' _type='${body._type}'`);

  return new Response(null, { status: 200 });
}
```

**Key implementation details:**
- Uses `webhookSecret` from `env.ts` (not `process.env` directly) — per D-04, consistent with project pattern
- Guards `isValidSignature !== true` (not `=== false`) — per Pitfall 4, `parseBody` returns `null` when no secret provided, which is distinct from `false`. Treating `null` as failure prevents accepting unsigned requests.
- All response bodies are `null` — per D-08, no error messages leaked in response
- Console logs use `[revalidate]` prefix — per D-09, key events logged for Vercel function log debugging
- `revalidateTag(tag, { expire: 0 })` — per research, Next.js 16 deprecates single-arg form. `{ expire: 0 }` forces immediate cache expiry.
- `trainingProgramsPage` is intentionally NOT in `TYPE_TAG_MAP` — per Pitfall 6, no loader fetches with that tag. The listing page uses `getAllTrainingPrograms` which is tagged `'trainingProgram'`, so publishing a `trainingProgram` already invalidates the listing page's data.
